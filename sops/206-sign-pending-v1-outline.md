# SOP 大纲 · 206 待签约跟进（v1 占位）

> **状态**：大纲，供运营/管家负责人填写。  
> **适用**：`status=206`，`updateTime` 在最近 14 天内，归属管家（`exts.supervisorId`）。  
> **不适用**：204 上门未成交、超过 14 天未更新的 206、已签约但未改状态（待数据规则识别）。

## 1. 目标

推动工单 **离开 206** → 签约或明确丢单原因，避免无效骚扰。

## 2. 建议动作枚举（Action Spec 桶）

引擎输出 `suggested_action` 时优先落入下列之一（便于统计采纳）：

| 动作码 | 说明 |
|--------|------|
| `CALL_SIGN` | 电话沟通签约意向与顾虑 |
| `VISIT_AGAIN` | 预约二次上门（方案/demo/勘漏） |
| `PRICE_TALK` | 议价/分期/对比竞品 |
| `COORD_PROPERTY` | 协调物业/邻里等外部因素 |
| `QUOTE_REVISE` | 修订报价单后再次推送 |
| `CLOSE_LOST` | 判定丢单并备注原因（仍须人工确认） |

- **识别信号**：`describe` 为空
- **建议动作**：`CALL_SIGN` 了解现场结论
- **避免**：假设已签约或已丢单

## 4. 负向清单（禁止建议）

- 不要建议管家跟进 **204** 工单
- 不要建议对 **超过 14 天未更新** 的单做重度投入
- 不要代替管家 **直接承诺降价/工期**（仅可建议「申请」）
- 不要 **自动发送** 给客户（仅管家审批后执行）

## 5. 与引擎字段映射

| 引擎字段 | SOP 用法 |
|----------|----------|
| `task_type` = 跟进签约 | 场景 3.x 上下文 |

| `event_type` = `STALE_SIGN_PENDING` | 固定 206 停滞事件 |
| `summary` / `title` | 场景 3.1、3.5 关键词 |
